Case Study

Meow Management

Restaurant operations, centralized.

React Native Expo TypeScript Supabase (Postgres + RLS) Claude (Anthropic API)
Github

The Problem

Small restaurant teams often track inventory counts, shift notes, and finances across disconnected paper logs, group texts, and spreadsheets. Meow Management centralizes those three things into one mobile dashboard, so a shift lead can check stock, log a note, or review finances from the same app.

My Role

I built Meow Management's mobile client and backend end-to-end: the Expo/React Native app, the Supabase schema and row-level security policies, the multi-tenant restaurant/team model, and Miso, an in-app assistant backed by Claude.

Key Technical Decisions

  • Supabase for both auth and data. Row-level security policies key every table — inventory, finance, tasks, activity log — to the authenticated user and their restaurant, so the same backend that stores the data also enforces who's allowed to touch it.
  • A real team model, not just a shared login. restaurants, restaurant_members, and restaurant_invites are separate tables. Invites are token-based, validated against the invited email, expire, and assign a role (owner vs. staff) on acceptance instead of just adding a name to a list.
  • Miso as a genuine Claude-backed assistant. "Ask Miso" calls Claude through the Anthropic API from a Supabase Edge Function, passing the caller's own RLS-scoped inventory, finance, and task data as context — and redacts raw user IDs to display names before anything reaches the model.
  • Manager approvals plus an append-only audit trail. Staff submit finance entries as requests that an owner or manager approves through a SECURITY DEFINER RPC, while an activity log written only by database triggers — never by clients — tracks who changed what.

Challenges & Solutions

Two real issues shaped the current data model, both caught by re-reading my own schema rather than a live incident. Early inventory, finance, and log tables shipped with an open "anon full access" policy — anyone holding the public anon key could read or write any row — which I replaced with row-level security policies scoped to the authenticated user. Later, once team invites shipped, a current_restaurant_id() helper that assumed one restaurant per user (limit 1) started returning the wrong restaurant for anyone who belonged to two; I fixed it by automatically retiring a user's empty, auto-created solo restaurant the moment they accept an invite to a real team.

Status & What's Next

Meow Management runs end-to-end against a live Supabase backend: authentication, inventory/task/finance CRUD, the Claude-powered Miso assistant, manager approvals, an audit trail, and email-based team invites all work today. Low-stock, overdue-task, and approval push notifications are wired through a device-token registry and a dispatch Edge Function.
